# 076 — Audit Trail: audit_events Table + logAudit() Utility (Phase 3)

**Date:** 2026-03-19
**Impact:** Compliance + operational visibility. Append-only audit trail for all admin actions. No behavior change for existing users.
**Gaps resolved:** GAP-22 (incomplete audit trail), GAP-16 (provisioning audit), DISC-02 (desisted_by tracking), DISC-05 (assignment audit logging).

## Why

No record existed of who performed admin actions — confirming reservations, rejecting them, inviting salespeople, changing rates, or modifying project assignments. If a reservation was incorrectly confirmed or a salesperson invited by mistake, there was no way to determine who did it or when. This is both an operational risk (Pati needs accountability) and a compliance gap (SOC 2 CC7.2 requires audit trails for privileged actions).

## What Changed

### New: `audit_events` Table (Migration 041)

Append-only table storing every admin action:

- `id` (uuid v7), `event_type`, `actor_id`, `actor_email`, `actor_role`
- `resource_type`, `resource_id`, `resource_label`
- `details` (JSONB — action-specific metadata)
- `ip_address`, `user_agent`, `created_at`

**RLS policies:**
- SELECT: admin roles only (master, torredecontrol) via `jwt_role()`
- INSERT: service_role only (server-side `logAudit()` uses admin client)
- UPDATE/DELETE: nobody (append-only)

### New: `src/lib/audit.ts`

Fire-and-forget `logAudit(user, event)` utility. Extracts actor identity from Supabase User object. Captures IP and user-agent from the request. Non-blocking — audit failures are logged to console but never block the API response.
